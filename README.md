# fraud-monitoring-system
Project Overview

This project implements a fraud detection service built around a machine learning model. The goal of the system is to simulate how a fraud detection model would operate in a real production environment.

Instead of just training a model, the focus of this project is on the full lifecycle of a machine learning system. The model is exposed through a REST API, containerized using Docker, deployed using CI/CD pipelines, and monitored to detect possible data drift.

The system allows users to send transaction data to an API endpoint and receive a fraud prediction in real time. In addition to predictions, the system also tracks usage statistics and provides basic monitoring capabilities.

Problem Motivation

Fraud detection is a common use case for machine learning in financial systems. However, in real applications the challenge is not only building the model but also deploying and maintaining it reliably.

A production system must:

Serve predictions through an API

Track model usage

Monitor model behavior

Detect when data patterns change

This project focuses on these operational aspects of machine learning systems.

API Endpoints

The service exposes several REST endpoints that allow interaction with the model.

Health Check
GET /health

Used to verify that the API service is running.

Example response:

{
  "status": "ok"
}
Model Information
GET /model-info

Returns basic information about the deployed model.

Example response:

{
  "model_name": "fraud_detection_model",
  "version": "1.0"
}
Fraud Prediction
POST /predict

This endpoint accepts transaction data and returns a prediction indicating whether the transaction is likely fraudulent.

Example request:

{
  "features": [0.15, 0.72, 0.03, 0.91]
}

Example response:

{
  "fraud_probability": 0.04,
  "threshold": 0.5,
  "prediction": 0
}

Where:

fraud_probability represents the probability that the transaction is fraudulent

threshold represents the classification threshold

prediction is the final result (0 = legitimate, 1 = fraud)

Statistics Endpoint
GET /stats

Returns basic usage statistics about the prediction service.

Example response:

{
  "total_predictions": 124
}
Reset Statistics
POST /reset-stats

Resets the prediction counters maintained by the service.

Dashboard Endpoint
GET /dashboard

Provides monitoring information related to the system.

Running the Project Locally
Install dependencies
pip install -r requirements.txt
Start the API server
uvicorn app.main:app --host 0.0.0.0 --port 8080

Open the API documentation:

http://localhost:8080/docs

FastAPI automatically generates interactive API documentation where endpoints can be tested directly.

Running with Docker

Build the Docker image:

docker build -t fraud-monitoring-system .

Run the container:

docker run -p 8080:8080 fraud-monitoring-system

The API will be available at:

http://localhost:8080/docs
CI/CD Pipeline

The project uses GitHub Actions to automate testing and deployment.

Three workflows are implemented:

1. CI Workflow

Runs whenever code is pushed to the repository.

This workflow:

installs dependencies

runs tests

validates the application

2. Deployment Workflow

Builds the Docker container and deploys the application to Google Cloud Run.

This ensures that updates to the repository automatically update the deployed API.

3. Monitoring Workflow

Runs on a schedule and executes monitoring scripts to detect possible data drift.

The workflow generates reports that help determine if the model should be retrained.

Monitoring and Drift Detection

Machine learning models can lose accuracy over time if incoming data changes. This phenomenon is known as data drift.

To address this, the project includes a monitoring script that periodically analyzes incoming data and generates drift reports.

These reports help determine when the model may need retraining.

Deployment

The API is deployed using Google Cloud Run, which allows the containerized application to run without managing servers.

Cloud Run automatically scales based on incoming requests and can scale down to zero when idle.
